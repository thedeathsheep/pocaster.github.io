提到嵌入检索，很容易联想到一整套图谱：向量库、重排、混合检索、Agent 编排……对小团队或个人项目来说，**第一步往往不是选型，而是证明「这件事值得存在」**：没有它，问题是否仍然可忍；有了它，错误是否可接受。

我后来把「最小可行」压到四件事：**切片、嵌入、存储、评测**。其余是增量。

![Minimal embeddings](/assets/img/roadmap.jpeg)

## 1. 切片：粒度比模型更重要

切片太细，上下文碎，语义不完整；切片太粗，噪声多，命中像抽奖。我的实用起点是：

- 按**标题层级**与**自然段**组合，而不是固定 512 token 一刀切；
- 技术文档保留**代码块与列表**在同一 chunk，避免「只有半段例子」；
- 对表格与配置，**整表一个 chunk** 或按行键拆分，视查询模式而定。

没有银弹，只有「最常见的十种问法」反推形状。

## 2. 存储：从内存到 SQLite 的阶梯

数据量小时，**启动时加载 numpy 数组 + id 映射**都能撑很久。再往上：

- **SQLite + 向量扩展**（若环境允许）降低运维面；
- 或托管向量服务，把精力留给数据管线。

我会刻意推迟分布式方案，直到单机顶不住——否则调试成本会先顶不住。

## 3. 检索闭环：先 top-k，再谈重排

第一版只要：

```text
query -> embed -> cosine top_k -> return text + score
```

若 top-k 经常漏答案，问题多半在切片与元数据，不在重排模型。把重排提前上，常常只是**把系统性错误打扮得更自信**。

## 4. 评测：几十条黄金问句胜过主观「感觉挺准」

我会准备三类问句（每类十来条就够起步）：

- **直述型**：答案几乎原文出现；
- **改写型**：同义说法、口语化；
- **否定型**：应返回「不知道/未覆盖」而不是胡编。

指标不用复杂：**Top-3 命中率**、**空结果率**、**明显胡扯率**（需要人眼或简单规则）。每周文档更新后跑一遍 diff。

## 5. 元数据：便宜但救命

给 chunk 加 `source_path`、`section_title`、`last_updated`，检索失败时才能回答「为什么没命中」。没有元数据，向量空间只是一团沉默的点。

## 6. 收束

小团队做嵌入检索，最危险的陷阱是**工具链先于问题定义**。当我把切片、存储与评测压到最小闭环，再决定是否引入重排与 Agent，路线会清晰很多：**先让检索诚实，再让生成说话**。
