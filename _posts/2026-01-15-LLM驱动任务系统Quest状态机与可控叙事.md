把 LLM 放进游戏任务系统最容易失败在两点：状态不可控、叙事不可验证。本文给一个“可控叙事”的结构：用确定性的 Quest 状态机管理进度，让 LLM 只负责生成“该状态下的文本与动机”，而不是直接改游戏状态。

![Quest system](/assets/img/game01.jpg)

## 1. 目标：让 LLM 只做“写作”，别做“裁判”

建议把系统拆成两层：

### 1.1 Deterministic layer（确定性层）
- Quest 状态迁移
- 奖励/失败条件
- 角色与世界变量（world state）

### 1.2 Generative layer（生成层）
- 在当前状态下生成描述（narration）
- 生成“下一步提示”（hint）
- 生成对话内容（dialogue snippets）

LLM 输出始终落在生成层：它不能直接改世界变量。

## 2. Quest 状态机：把“进度”写死

<div class="mermaid">
stateDiagram-v2
  Draft --> Offered: create
  Offered --> Accepted: accept
  Accepted --> InProgress: start
  InProgress --> Completed: condition met
  InProgress --> Failed: fail condition
  Completed --> Archived: reward collected
  Failed --> Archived: handled
</div>

### 2.1 每个状态的生成任务
- Offered：生成任务简介与吸引点
- Accepted：生成任务动机与目标清单
- InProgress：生成阶段提示、风险提醒与可选支线
- Completed：生成完成叙事与奖励呈现文案
- Failed：生成复盘叙事与“下一次尝试”引导

这样 LLM 的输出会被强约束在“当前状态的语义范围”里。

## 3. 可控叙事：用结构化输出而不是自由发挥

建议让 LLM 以结构化方式输出“文本模块”，例如：
- `title`
- `narration`
- `objective_list`
- `hint`
- `dialogue_preview`

然后在客户端/脚本层做渲染与变量填充。

### 3.1 约束的关键：禁止 LLM 直接推理世界

Prompt 的规则可以很明确（伪代码）：

```text
Rules:
  - Do not change quest variables.
  - Only describe what is consistent with given world_state.
  - If info is missing, ask a clarifying question.
Output schema:
  title: string
  narration: string
  objective_list: list[string]
  hint: string
```

## 4. 评估：如何判断 LLM 的“叙事质量”达标？

建议分三类评估：

- Grounding：生成内容是否与世界变量一致
- Utility：提示是否能帮助玩家做行动选择
- Style consistency：角色/阵营语言风格是否一致

我会对每次上线做回归：同一状态同一 world_state，生成内容是否仍然一致可信。

## 5. 最小可行实现（MVP）

从最小版本开始：
1. 固定 3 个状态（Offered / Accepted / InProgress）
2. LLM 只生成 `narration` 与 `hint`
3. 客户端渲染后做一次人工验收
4. 稳定后再扩展 `dialogue_preview` 与 `objective_list`

## 6. 下一步：把“玩法”也交给状态机

当状态机稳定后，我会进一步把支线、道具奖励、敌人刷新规则都显式化。届时 LLM 仍然只做写作：它让任务“更像一个世界的一部分”，而不是随机故事生成器。
