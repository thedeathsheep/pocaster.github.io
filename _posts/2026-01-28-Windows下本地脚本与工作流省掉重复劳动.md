很长一段时间里，我低估「重复五次以上的手工活」的毒性：每次只要三分钟，但会打断心流，还会因为人肉失误留下诡异 bug。后来我在 Windows 环境里养成了一个习惯：**第三次手做同样的事，就允许自己写十行脚本**——不必漂亮，必须可重复。

![Local workflow](/assets/img/path.jpg)

## 1. 先分清：一次性、偶发、节律性

不是所有事都值得自动化。

- **一次性迁移**：写脚本往往不如手动点完。
- **偶发且高风险**：脚本要写得很严谨，成本可能高于手工。
- **节律性重复**：备份、日志归档、批量重命名、生成缩略图、拉取多仓库——这是脚本甜点区。

我对自己的判断标准是：**若这件事下周还会以同样形状出现，就进入候选**。

## 2. PowerShell：默认装在系统里就是优势

在 Windows 上，PowerShell 的最大优势不是语法优雅，而是**到处都有**。我常用它做：

- `Get-ChildItem` 批量筛选扩展名；
- `Copy-Item` / `Robocopy` 做增量同步；
- 简单的 JSON / CSV 读写处理构建产物。

脚本开头我会写清：

- 工作目录如何定位（相对脚本路径还是显式参数）；
- 失败时是否 `exit 1`（方便 CI 或任务计划程序识别）。

## 3. 小约定：一个 `tools/` 目录胜过十个散落 `.ps1`

个人项目里我会放：

- `tools/`：可版本管理的脚本；
- `local/`（gitignore）：机器相关路径、密钥占位说明。

这样换电脑时，至少知道**哪些东西该从仓库恢复、哪些该本地补**。

## 4. 计划任务：让「记得跑」变成「系统跑」

对周更类任务（清理下载目录、打包日志、同步网盘），Windows 任务计划程序足够用。我会注意：

- 触发器是否与笔记本合盖睡眠冲突；
- 失败时是否写日志文件；
- 是否需要「只在交流电源时运行」这类条件。

自动化若静默失败，比不做更可怕。

## 5. 何时停手：别为脚本写脚本

见过最浪费的路径是：为了维护脚本，又写了一个配置语言，再写解析器。对个人工作流，**YAML/JSON + 50 行 PowerShell** 往往到顶。

停手信号是：脚本调试时间开始稳定超过手工时间的一半。

## 6. 与 MinGW / 工具链的关系

本地若装了 MinGW（常见路径如 `C:\mingw64\bin`），我会把它加入用户 PATH，但脚本里**不硬编码绝对路径**——改用环境变量或开头检测，避免换机即碎。C/C++ 构建仍交给 CMake 或现有工程脚本，PowerShell 只负责「调用与打包」，不抢编译器的活。

## 7. 收束

工作流自动化不是为了炫技，是为了**把注意力留给不可自动化的判断**。在 Windows 上，十行可靠的脚本，常常比一整套「未来可扩展」的框架更忠诚。土办法能活下来，通常是因为真的省时间。
