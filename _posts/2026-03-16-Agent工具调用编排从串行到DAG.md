Agent 真正决定体验上限的，常常不是“模型好不好”，而是“工具调用怎么编排”。串行调用看起来简单，但在真实场景里会带来两类问题：延迟变长、失败影响范围变大。本文把工具调用视为一个可验证的执行图：用 DAG（有向无环图）来做并行化与可回滚。

![Tool orchestration](/assets/img/AI-Agent-Workflow.png)

## 1. 为什么从串行到 DAG？

假设一个任务需要：
- 查询知识库（KB）
- 调用业务工具（TicketCreate）
- 再根据 KB 结果做校验（Validation）

串行写法通常像这样：

```text
KB -> Tool -> Validation -> Reply
```

当其中某些步骤互相独立时，DAG 就能把等待时间“切开”，同时让失败可控。

## 2. 工具调用执行图（DAG）的最小定义

<div class="mermaid">
graph TD
  A[KB Search] --> B[Policy Validate]
  A --> C[Tool Call]
  B --> D[Compose Answer]
  C --> D
</div>

你需要的不是“完美图”，而是清晰的三样东西：
- 节点：某个工具调用 / 某个校验步骤
- 边：依赖关系（谁必须等谁）
- 结果：每个节点输出是什么，以及失败时怎么标记

## 3. DAG 节点的输出契约：让调度器可验证

建议每个节点都返回结构化结果，而不是仅返回文本：

```json
{
  "node_id": "tool_ticket_create",
  "status": "success",
  "output": {
    "ticket_id": "T-1024"
  },
  "error": null,
  "trace": {
    "duration_ms": 1200
  }
}
```

这样调度器才能做两件事：
- 判断依赖是否满足（依赖节点 success 才可继续）
- 失败时执行降级/重试策略（并且可观测）

## 4. 并行化策略：哪些步骤适合拆成 DAG？

经验上，适合并行的通常满足：
- 输入只依赖同一个上游结果
- 输出彼此不互斥
- 有清晰的失败处理方式

例如：
- 检索（KB Search）可以同时跑多个 query（做汇总）
- 风险校验（Policy Validate）可以并行准备
- 不同候选证据可以并行 rerank 再合并

## 5. 回滚与降级：失败不必“一刀切”

DAG 最重要的产品意义是：失败范围收敛。

你可以设置三种失败模式：
- Soft fail：允许缺失部分证据，仍可回答（但会降低 groundedness）
- Hard fail：关键工具失败，必须追问或转人工
- Retry：对超时/网络类失败重试（避免无意义回问）

## 6. MVP：从两层 DAG 开始

为了快速落地，建议从最小 DAG 开始：
1) 上游：KB Search / policy gate 并行准备
2) 下游：基于结果决定是否调用业务工具
3) 最后：统一生成回答并附带 trace

当两层跑稳，再扩展到多分支与条件执行。

## 7. 最后：编排能力是一种“产品能力”

对用户来说，最直观的收益是：
- 更低延迟
- 更少“凭空回答”
- 更可控的失败体验（追问/转人工更合理）

工具集合（SearchKB、SQL、工单、CRM 等）越长，DAG 越要克制：节点多不等于能力强，**清晰依赖 + 可观测失败** 才是能长期运维的图。具体画成什么样，取决于你的业务链路，但上面的契约与回滚思路可以一直复用。
