很长一段时间里，我把「手感」当成一种神秘天赋：厉害的策划拧两下参数就对味了，不厉害的怎么改都像在泥里走路。后来参与的项目规模变大、平台变多，**纯靠肌肉记忆调参开始不够用**——版本一多，谁也说不清「上周那个对味的版本」到底改了哪几个数。

我现在的看法更朴素：**手感是输入与反馈在时间轴上的对齐问题**。对齐了，玩家觉得「跟手」；没对齐，再华丽的特效也像在延迟演戏。把这句话落实，就需要把关键时间窗与状态迁移记录下来，让调参变成可复盘的数据工作。

## 1. 先画「时间轴」，再谈数值

对多数动作或 ARPG 战斗，我会先列一条**玩家动作时间轴**与一条**敌人反馈时间轴**，看它们如何咬合。关心的不是单一帧，而是区间：

- 从按下按键到角色开始执行（输入延迟 + 动画切入）。
- 从执行开始到判定框有效（startup）。
- 判定有效的持续帧（active）。
- 收招到可再次行动（recovery / endlag）。

敌人侧对称地有：预警、前摇、受击框、反击窗、无敌或霸体窗口。

当这些区间在文档里还是模糊词（「稍微快一点」）时，协作成本会爆炸。把它们落成表（哪怕先在表格里），沟通效率会立刻上去。

## 2. 输入缓冲与预输入：「跟手」的第一道门

### 2.1 输入缓冲（input buffer）

缓冲允许在动作尚未结束时提前登记下一次输入，在可衔接帧到来时自动执行。缓冲窗口太短，玩家觉得系统聋；太长，又容易串招误触。

我习惯为不同动作类型设**不同缓冲上限**（轻攻击、重攻击、闪避、技能），而不是全局一个常数。

### 2.2 预输入（queue / leniency）

与缓冲接近但语义略有差别：预输入更强调「在严格窗口边缘的宽容」。这里最适合做 A/B：同一关卡片段，调整 2～3 帧，观察通过率与重试率。

## 3. 无敌帧、霸体与受击硬直：「公平感」来自可读性

闪避无敌帧（i-frames）过长，战斗变无聊；过短，玩家觉得被阴。关键不一定是绝对长度，而是**敌人预警是否与无敌窗对齐**：玩家能否在 fair 的条件下做出反应。

受击硬直（hitstun）与击退则影响节奏密度。我会把它们与**敌人攻击间隔**、**玩家恢复帧**放在同一张表里对照，而不是孤立调一个常数。

霸体（super armor）要配合**可识别的视觉语言**：否则玩家会以为自己在输出窗口里，实际却在帮敌人按摩。

## 4. 伤害期望与资源循环：手感不只是帧数据

TTK（time-to-kill）与资源获取（体力、蓝条、架势条）会改变同一套帧数据的「主观难度」。我习惯为每个精英模板维护一张**期望表**：

- 平均击杀时间区间；
- 玩家需要执行的「有效操作次数」；
- 失误一次的惩罚（掉血比例、硬直长度、是否打断资源循环）。

当帧数据调顺了但玩家仍喊「刮痧」或「被秒」，问题往往在这张期望表上，而不是某一帧多了一帧。

## 5. 遥测：把战斗过程变成可聚合事件

没有遥测，复盘只能凭记忆。我会尽量记录**低侵入、高语义**的事件（示意）：

```text
combat_encounter_start { encounter_id, enemy_archetype, build_hash }
player_action { frame, action_type, buffered: bool }
hit_confirm { frame, damage, was_crit, enemy_state }
player_damage_taken { frame, source, remaining_hp_pct }
encounter_end { outcome, duration_frames, retry_count }
```

字段名与粒度按项目裁剪，原则是：**能还原「玩家在关键窗口里做了什么」**，而不是只记胜负。

## 6. 「第二套关卡」：用曲线对比版本

把一次遭遇战的关键指标画成曲线（帧或时间为横轴），例如：

- 累积伤害输出；
- 玩家位置风险（与危险区的距离分布）；
- 输入密度（每秒有效输入次数）。

版本对比时，不是只看平均值，而是看**分布尾部**：p95 遭遇时长、p95 连续受击帧数。尾部变差，往往先于均值变差暴露设计问题。

## 7. 与关卡设计的接口：战斗不是孤岛

同一套战斗参数，在不同关卡几何里手感会变窄道、高低差、视野遮挡都会改变「可读性」。所以我会要求关卡导出**几何标签**（开阔 / 狭窄 / 垂直压力）并在遥测里带上，否则策划会陷入「到底是人有问题还是关有问题」的争吵。

这与我在「AI 辅助关卡设计」那篇里强调的**可验证参数**是同一逻辑：战斗手感也要能对照关卡上下文解释。

## 8. 收束

手感调参最怕两件事：**不可记录**与**不可对比**。当我把时间轴、窗口、期望与事件日志接起来之后，讨论会从「再软一点」变成「把缓冲从 6 帧试到 8 帧，看 p95 重试是否下降」。这不是消灭直觉，而是给直觉一条跑道——**直觉负责提假设，数据负责验假设**。
