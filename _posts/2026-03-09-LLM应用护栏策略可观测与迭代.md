「加个护栏」在会议里出现的频率很高，但落地时常常被简化成：**在 system prompt 里多写几句不要做什么**。我经历过的问题是：prompt 越来越长，行为却越来越不可解释——有时管用，有时像抽签；一出事只能全员加班试复现。

我现在的做法是：把护栏当成**策略系统**，和 prompt 解耦一部分。策略要有版本、有优先级、有可观测事件，最好能离线回放。否则「安全」只是气氛组。

## 1. 护栏到底在防什么？

先列目标，再选手段。常见目标包括：

- **合规与品牌**：仇恨、色情、违法指导、泄露隐私。
- **事实与引用**：无检索时禁止冒充权威；有检索时必须可追溯到片段。
- **工具滥用**：SQL 只能在只读副本上执行；写操作必须二次确认。
- **成本与滥用**：超长上下文、疯狂重试、批量自动化调用。

不同目标的失败模式不同，混在一段「道德声明」里很难维护。

## 2. 四层护栏：输入 / 生成 / 输出 / 工具

### 2.1 输入侧（Input guards）

在进模型前处理：

- 速率限制、身份鉴权、租户隔离。
- PII 检测与脱敏（邮箱、电话、证件号的处理策略写清楚）。
- 明显的注入模式拦截（「忽略以上规则」这类签名库可以维护一份）。

输入侧的优势是**便宜、确定性强**；缺点是只能看到文本表面，不懂语义。

### 2.2 生成侧（Generation policy）

这里才是 prompt、温度、模型选择、以及是否启用 schema-guided decoding 的组合拳。我会把「软约束」尽量转成「硬形状」：能 schema 化的不留在自由文本里。

### 2.3 输出侧（Output guards）

模型吐出来之后：

- 二次分类器或规则检测（视场景与预算而定）。
- 对最终用户可见内容与对工具传递内容分开处理（有时需要不同严格度）。

输出侧适合拦截「看起来礼貌但其实越界」的情况，但要警惕**延迟与误杀**。

### 2.4 工具侧（Tool guards）

最容易被忽略，也最容易出大事：

- 参数 schema 校验。
- 权限白名单（哪个 key 能调哪个工具）。
- 幂等与超时（避免重试雪崩）。

## 3. 策略即数据：版本、优先级、灰度

我会把护栏规则当成配置，而不是散落在十三个文件里的 if：

- **`policy_id` + `policy_version`**：与模型版本、prompt 模板版本一起写入日志。
- **优先级与短路**：明确「哪条规则先执行」「命中后是否还跑后续规则」。
- **按租户/场景灰度**：新规则先对小流量开启，对比误杀率与违规拦截率。

没有版本信息，线上一旦出现「昨天还好今天不行」，基本只能猜。

## 4. 可观测：把「拦截」变成事件，而不是 print

每次拦截或降级，我尽量记录结构化事件（示意字段）：

```text
event: guardrail_triggered
policy_id: output_pii_block
policy_version: 20260328.3
reason_code: PII_EMAIL
route: user_visible_reply
action: redact_and_continue | block | escalate_human
latency_ms: 12
request_trace_id: ...
```

这样仪表盘上可以回答：

- 哪条规则触发最多？
- 误杀投诉是否集中在某版本之后？
- 输出侧护栏是否把 p95 延迟顶穿 SLA？

## 5. 离线回放：用日志养测试集

我会定期从日志抽样构建 **red team / regression set**：

- 已知违规样本应被拦截。
- 已知正常样本不应被误杀。
- 边界样本（讽刺、虚构、游戏内暴力描写）按产品政策标注期望行为。

每次改 policy，先在回放集上跑 diff，再谈上线。否则只能靠用户当测试员。

## 6. 与 RAG、Agent 的衔接

若系统里有检索（RAG）或多步 Agent：

- **检索护栏**：哪些库可见、哪些文档版本有效、引用格式是否强制。
- **Agent 护栏**：最大步数、最大工具调用次数、单步超时、失败累计阈值。

这些限制写在架构里，比写在 prompt 里可靠。prompt 适合表达语气与任务描述，不适合承载「分布式系统的纪律」。

## 7. 我会避免的误区

- **只靠加长 system prompt**：成本高、漂移快、难回归。
- **把「模型拒绝」当安全证明**：拒绝理由与真实风险不是一回事。
- **忽视工具层**：文本再干净，工具参数一样可以作恶。
- **没有误杀指标**：只有拦截率没有体验指标，产品会被自己勒死。

## 8. 小结

护栏的本质是**在不确定性周围修筑可验证的边界**。我的实践顺序一般是：

1. 先列出真实事故类型，而不是抽象「AI 安全」；
2. 输入/输出/工具分层落地，每层有明确 owner；
3. 策略版本化 + 事件化观测；
4. 离线回放集锁住回归。

做到这一步，prompt 反而可以 shorter、更专注任务本身——因为纪律已经住在系统里了。
