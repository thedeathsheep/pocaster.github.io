游戏里的“AI 角色”最难的是一致性：今天像一名冷静的顾问，明天却变成了热血队友。本文讨论一种实用结构：把记忆分层，把约束写进生成阶段，并用可回放的数据验证一致性。

![Character memory](/assets/img/yimei.png)

## 1. 一致性从哪里来？

角色不一致往往不是“模型抽风”，而是缺少明确的输入：
- 角色是谁（Identity）
- 角色在这个世界里的立场（Belief）
- 这段对话发生了什么（Conversation state）
- 需要遵守的规则（Constraints）

## 2. 记忆分层怎么用：把长短期责任分开

### 2.1 Identity profile（长期）
包括：
- 角色的目标、底线、说话习惯
- 语言风格（短句/长句、是否用口头禅）
- 禁忌主题列表（避免跑偏）

### 2.2 Session memory（短期）
包括：
- 最近对话的关键事实（用于承接）
- 玩家提问中缺失的信息（用于澄清）

### 2.3 World stance（世界立场）
包括：
- 阵营立场与对事件的态度
- 与任务相关的变量（例如：当前任务进度）

## 3. 约束生成：让模型知道“该停在哪里”

除了写 Prompt，还建议把约束当作结构化规则处理，例如：

```text
Constraints:
  - Must address the current objective.
  - Must not contradict identity profile.
  - Must not propose actions outside allowed tools.
  - If player intent is unclear, ask 1 clarifying question.
Output:
  - dialogue_text (string)
  - attitude_tags (list[string])
```

客户端渲染时可以用 `attitude_tags` 做语气控制。

### 3.1 Prompt 模板：约束优先

下面给一个“可复制”的约束优先模板（我会把字段映射成项目的实际结构）：

```text
Identity:
  role_name: "Scout"
  style: "short sentences"
  taboo: ["lie", "predict future"]

World stance:
  faction: "Neutral"
  belief: "Player intent is uncertain"

Conversation state:
  last_player_message: "{{last_user}}"
  missing_info: ["goal", "constraints"]

Constraints:
  - If missing_info exists, ask 1 clarifying question.
  - Do not contradict taboo list.

Output:
  dialogue_text
  attitude_tags
```

## 4. 一致性验证：可回放的评估方式

建议做两类评估：

- Replay eval：同一角色、同一世界变量、同一对话历史，输出是否稳定一致
- Contrast eval：同一角色不同阵营/不同任务阶段，输出是否符合预期差异

只要输入是可回放的，我就能用自动化回归让一致性“可经营”。

## 5. MVP 建议：先保证“语气一致”，再做“内容一致”

推荐最小可行路线：
1. 固化 Identity profile（先别做太复杂）
2. 用约束让模型不自相矛盾
3. 先做到“语气与立场一致”
4. 再扩展 Session memory，让承接更自然

## 6. 最后：一致性是产品能力，不只是提示词

当角色一致性变成“有数据、可回放、可回归”的工程能力，才算真正把 LLM 变成游戏系统的一部分。

实际落地时，Identity profile 的字段不必一开始就求全：先把阵营、禁忌、说话习惯钉死，再逐步加任务阶段与记忆承接。一致性是迭代出来的，不是一次 prompt 写满就结束。
