把 AI 用在游戏对话上，最容易翻车的不是“写得不够好”，而是“可玩性不可控”：分支过密、触发条件不一致、叙事节奏断裂，甚至形成不可达路径。本文给一个面向工程的思路：让 AI 生成“对话树节点”，而不是直接生成最终脚本；同时用可验证约束让对话树不会跑偏。

![Dialogue tree](/assets/img/yimei.png)

## 1. 把对话树拆成可验证的数据结构

建议把节点表示为：
- 文本：这句话要怎么说
- 条件：什么时候能触发
- 转移：选择哪个选项会进入哪里
- 约束：这条节点必须满足的叙事/玩法规则

## 2. AI 的职责：生成“节点提案”，交给校验器落地

实践中建议用“两步式管线”：

### 2.1 Generation（生成）
AI 只生成“节点提案”（node draft），包括条件、对话文本、选项列表、下一节点 id。

### 2.2 Validation（验证）
校验器检查：
- 条件是否自洽（不能说“我已经完成了任务”，却又没给任务状态）
- 是否存在不可达节点（避免死分支）
- 节奏是否满足目标（例如关键节点不能挤在相邻两轮）

## 3. 约束清单：让对话树不再“玄学”

下面是常用的可玩性约束（我会按项目简化）：

- Reachability：从入口节点必须能到达关键节点
- No-Contradiction：对话状态与世界变量一致
- Pace Budget：关键节点之间最小间隔
- Choice Budget：每轮选项数量上限（例如 <= 4）

![Pacing constraints](/assets/img/game01.jpg)

## 4. 可验证输出 schema（示例）

建议让 AI 输出结构化 schema，便于验证与回放：

```json
{
  "node_id": "quest_offer_01",
  "condition": {
    "quest_state": "NotStarted",
    "player_level_min": 3
  },
  "dialogue_text": "我愿意帮忙。",
  "choices": [
    { "text": "接受", "next_node": "quest_accept_01" },
    { "text": "再看看", "next_node": "quest_greeter_01" }
  ],
  "constraints": [
    { "type": "PaceBudget", "min_turn_gap": 2 }
  ]
}
```

## 5. MVP：先做一棵小树，保证“可达 + 自洽”

最小可行路线：
1. 入口节点 + 关键节点（至少 1 个任务起点）
2. 每轮最多 2-3 个选择
3. 先不追求无限分支，让玩家能“看懂并做决定”
4. 用自动化校验器跑 reachability 与 contradiction

## 6. 最后：可玩性是工程约束，不是写作灵感

当我把“可玩性”变成可验证的约束，对话树就能持续迭代，而不是一次性写死。AI 在这里的价值是加速产出节点提案，让我把时间花在最关键的叙事决策上。
